# CZ1007 재도전용 공부 체크리스트

## 목적
- [001_initial-evaluation.md](/Users/sijun-yang/Documents/GitHub/data_structures_docker/reports/001_initial-evaluation.md)에서 `Fail` 또는 `Warn`가 나온 문제를 다시 풀기 전에, 어떤 개념을 먼저 보강해야 하는지 빠르게 점검하기 위한 문서다.

## 1. 문제 해석과 조건식 변환

### 왜 먼저 봐야 하나
- 현재 Stack and Queue 1~3은 재평가 기준으로 핵심 오답 단계는 벗어났다.
- 이 섹션은 비슷한 실수를 다음 문제에서 반복하지 않기 위한 일반 체크 용도로 남긴다.

### 공부 체크
- 문제 문장을 읽고 “남겨야 하는 값”과 “제거해야 하는 값”을 따로 적을 수 있는가
- 함수 이름과 메뉴 문구를 보고 조건이 서로 일치하는지 확인하는 습관이 있는가
- 예제 3개 정도를 손으로 먼저 시뮬레이션할 수 있는가
- dequeue/enqueue 중에 `size`가 변하는 구조에서 루프 횟수를 어떻게 고정해야 하는지 설명할 수 있는가

### 다시 풀 때 체크리스트
- 입력 예시를 2개 이상 직접 적는다
- 반복문 안에서 실제로 어떤 값이 남아야 하는지 주석 한 줄로 적는다
- 구현 후 `1,2`, `1,2,3`, `2,4` 같은 작은 케이스로 바로 검증한다
- 순회 시작 전 원래 길이를 따로 저장해야 하는지 먼저 확인한다

## 2. 트리 재귀 기본 패턴

### 왜 먼저 봐야 하나
- `Q1_E_BT - identical`
- `Q2_E_BT - maxHeight`
- `Q3_E_BT - countOneChildNodes`
- `Q8_E_BT - hasGreatGrandchild`
- 이 문제들은 전부 재귀 자체는 시도했지만, base case나 좌우 결과를 합치는 규칙이 틀린 경우다.

### 공부 체크
- 재귀 문제를 볼 때 아래 3단계를 항상 분리해서 생각하는가
- `base case`
- `left/right recursive call`
- `current node에서 결과 결합`
- 높이 문제에서 `max(left, right)`를 써야 하는지 `min(left, right)`를 써야 하는지 구분할 수 있는가
- 개수 세기, 합계 구하기, 최솟값 찾기, 구조 동일성 판정이 각각 어떤 반환값 패턴을 가지는지 설명할 수 있는가
- 둘 다 `NULL`일 때 참인지 거짓인지 먼저 결정할 수 있는가

### 다시 풀 때 체크리스트
- 함수 맨 위에 base case를 먼저 쓴다
- `left_result`, `right_result` 변수를 따로 둔다
- 마지막 return 문장이 문제 요구와 맞는지 말로 읽어본다
- 깊이 문제는 반드시 “더 깊은 쪽”인지 확인한다

## 3. 연결 리스트 포인터 조작

### 왜 먼저 봐야 하나
- `Q2_A_LL - alternateMergeLinkedList`
- `Q6_A_LL - moveMaxToFront`
- 이 문제들은 값 복사보다 노드 연결 갱신이 핵심인데, head 변경이나 prev 처리에서 흔들린 흔적이 있다.

### 공부 체크
- singly linked list에서 `prev`, `curr`, `next`의 역할을 구분할 수 있는가
- head가 바뀌는 경우 왜 `ListNode **`가 필요한지 설명할 수 있는가
- 노드를 새로 만들지 않고 기존 노드를 재연결하는 방식과 값만 복사하는 방식의 차이를 이해하는가
- “현재 최대값 노드가 이미 head인 경우” 같은 특수 케이스를 먼저 떠올릴 수 있는가

### 다시 풀 때 체크리스트
- 빈 리스트
- 노드 1개
- head가 정답인 경우
- tail이 정답인 경우
- 중간 노드가 정답인 경우
- 위 5개를 먼저 손으로 그린다
- 포인터 변경 순서를 종이에 화살표로 적고 구현한다
- head가 바뀌는 문제는 마지막에 `*ptrHead`가 무엇을 가리켜야 하는지 다시 확인한다

## 4. null / 빈 구조 방어

### 왜 먼저 봐야 하나
- `Q1_F_BST - levelOrderTraversal`
- `Q3_F_BST - preOrderIterative`
- 빈 트리/빈 구조를 먼저 처리하지 않으면, 정상 케이스는 맞아도 평가에서 감점되기 쉽다.

### 공부 체크
- 함수 시작 시 `if (root == NULL)` 또는 `if (head == NULL)`를 자동으로 떠올릴 수 있는가
- push/enqueue 전에 원소 자체가 유효한지 확인하는 습관이 있는가
- `NULL` 역참조가 어느 줄에서 나는지 추적할 수 있는가

### 다시 풀 때 체크리스트
- 빈 구조 입력을 첫 테스트로 넣는다
- 첫 push/enqueue 전에 null guard가 있는지 확인한다
- 출력 함수라면 “아무것도 출력하지 않고 종료”가 맞는지 문제 의도와 대조한다

## 5. 함수 시그니처와 반환 계약

### 왜 먼저 봐야 하나
- `Q6_A_LL - moveMaxToFront`
- 반환형이 `int`인데 값을 반환하지 않는 문제처럼, 구현 전에 함수 계약을 놓치면 맞는 로직도 감점된다.

### 공부 체크
- 함수 선언을 읽고 입력, 출력, 부수효과를 각각 따로 적을 수 있는가
- “이 함수는 값을 반환하는가, 포인터를 바꾸는가, 둘 다 하는가”를 구분할 수 있는가

### 다시 풀 때 체크리스트
- 구현 전에 함수 시그니처 아래에 한 줄 메모를 남긴다
- 반환형이 `int`면 성공/실패/결과값 중 무엇을 돌려줘야 하는지 먼저 정한다
- 모든 return path가 실제로 값을 돌려주는지 마지막에 확인한다

## 6. 함수 내부 메모리 책임

### 왜 먼저 봐야 하나
- `Q7_C_SQ - balanced`
- 바깥에서 정리해 줄 수 없는 지역 자료구조는 함수 내부 조기 반환에서도 책임이 남는다.

### 공부 체크
- malloc으로 만든 메모리를 누가 해제하는지 함수 단위로 설명할 수 있는가
- 조기 return 전에 정리 코드가 필요한지 판단할 수 있는가
- “외부에서 정리 가능”한 자원과 “이 함수 안에서만 존재”하는 자원을 구분할 수 있는가

### 다시 풀 때 체크리스트
- 함수 안에서 만든 임시 자료구조가 있는지 먼저 본다
- 중간 실패 return이 있다면 정리 없이 빠져나가는지 확인한다
- 가능하면 cleanup 경로를 하나로 모은다

## 7. 재시도 순서

### 1단계: 가장 먼저 다시 풀기
- `Q2_E_BT`
- `Q3_E_BT`
- `Q8_E_BT`
- `Q1_E_BT`
- 이유: 지금 남아 있는 핵심 오답은 트리 재귀 패턴 쪽이 우선이다.

### 2단계: 포인터 조작 복습 후 다시 풀기
- `Q2_A_LL`
- `Q6_A_LL`
- 이유: 연결 리스트 문제는 종이에 그려 보고 다시 시도하는 편이 훨씬 효과적이다.

### 3단계: 엣지 케이스 보강
- `Q1_F_BST`
- `Q3_F_BST`
- `Q7_C_SQ`
- 이유: 큰 로직보다 빈 입력, 내부 자원 정리, 안정성 보강 중심 문제다.

## 7-1. 현재 기준으로 통과 처리된 항목
- `Q1_C_SQ - removeOddValues`: 초기 `count`를 고정해 원래 queue 원소를 정확히 한 번씩 처리하도록 수정됐다.
- `Q2_C_SQ - removeEvenValues`: 짝수 삭제 로직은 현재 구현 기준으로 맞다.
- `Q3_C_SQ - isStackPairwiseConsecutive`: 이번 평가에서는 “원소 수가 홀수면 false” 해석을 채택했고, 그 기준에서 현재 구현은 맞다.
- `Q1_C_SQ - createQueueFromLinkedList`, `Q2_C_SQ - createStackFromLinkedList`: 재생성 시 중복 누적은 함수 자체보다 제공 `main` 프레임의 상태 관리 문제로 본다.

## 8. 공부할 때 최소 습관
- 구현 전에 예제 2~3개를 손으로 시뮬레이션한다
- base case를 먼저 적는다
- head/root가 비는 경우를 먼저 본다
- return 값 의미를 먼저 고정한다
- 구현 후에는 작은 입력으로 바로 확인한다
